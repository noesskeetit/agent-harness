# T4: Параллельные воркеры + лимит конкуренции

Контекст: T1-T3 приняты, сценарий A1 прошёл живьём. Драйвер умеет
spawn_worker (одиночный). Теперь параллельная пачка. Перечитай SPEC.md.

## Что сделать

### 1. Config (`config.py` — в этой задаче менять МОЖНО)
Новое поле `max_parallel_workers: int = 4`, env `SWARM_MAX_PARALLEL_WORKERS`.

### 2. Тул `spawn_workers` в loop.py

- Схема: `spawn_workers(tasks: array)` — каждый элемент объект
  `{task: string, worker_id?: string}`. Описание для модели: «запусти
  НЕЗАВИСИМЫЕ подзадачи параллельно; не используй для подзадач, где одна
  зависит от результата другой — для них последовательные spawn_worker».
- Исполнение: `concurrent.futures.ThreadPoolExecutor` с
  `max_workers=config.max_parallel_workers`. Каждый элемент →
  `spawn_codex_worker(...)`.
- worker_id: явный — как есть; пустой — автогенерация продолжением нумерации
  worker-NN с учётом существующих директорий И уже занятых id в этой же
  пачке. Дубликат id в пачке → ошибка тула (текстом, без исключения наружу).
- Результат тула: блоки по каждому воркеру В ПОРЯДКЕ ВХОДНОГО СПИСКА,
  формат блока тот же, что у spawn_worker, разделитель `\n---\n`.
  Исключение в одном воркере не валит пачку: его блок = `worker=<id>
  ok=false` + текст ошибки.

### 3. Тесты

`tests/test_worker.py` или новый `tests/test_parallel.py`:
1. Поведение параллельности: monkeypatch `spawn_codex_worker` на фейк,
   который спит 0.3с и возвращает WorkerResult(ok=True). 3 задачи,
   cap=4 → суммарное время < 0.7с; те же 3 задачи, cap=1 → время >= 0.9с.
2. Автонумерация: в run_dir уже есть workers/worker-01; пачка из двух задач
   без id → получают worker-02 и worker-03, оба блока в результате.
3. Порядок: фейк с разными задержками (0.3с / 0с) → блоки в результате
   в порядке входного списка, не в порядке завершения.
4. Один воркер кидает исключение → его блок ok=false с текстом, остальные
   ok=true, тул вернул строку, ран не упал.

### 4. Живой прогон
`uv run swarm run "запусти параллельно двух воркеров: первый создаёт в своей
директории файл alpha.txt с текстом 'альфа', второй — beta.txt с текстом
'бета'. Когда оба завершатся, проверь оба файла и заверши с перечислением
содержимого"`.
В финальном сообщении: stdout, ls обоих workspace, содержимое обоих файлов,
и подтверждение из transcript.jsonl, что был ОДИН вызов spawn_workers (а не
два spawn_worker).

2. Живой прогон из п.4 с фактическими выводами.
3. Один коммит `feat: parallel workers with concurrency cap`.
