# swarm-harness — спецификация ядра

Тонкий harness в духе Claude Code dynamic workflows / Kimi agent swarm:
пользователь даёт одну сложную задачу одной командой, LLM-драйвер сам
раскладывает её, делегирует воркерам, проверяет результат исполнением и
возвращает ответ.

Предыдущая итерация этой идеи (`~/kaggle_grinder`) умерла под весом
самосертификации: A2A-конверты, artifact packages, review gates, leases,
evidence ledgers. Этот проект — инверсия: **интеллект в модели, харнесс —
тупой и маленький**.

## Поведенческий критерий успеха (единственный)

```bash
uv run swarm run "сложная многошаговая задача"
```

→ система сама декомпозирует, исполняет воркерами, верифицирует исполнением,
возвращает результат. Человеческих вмешательств между командой и результатом: 0.

## Не-цели и анти-bloat правила (важнее архитектуры)

1. ЗАПРЕЩЕНО без явного решения оператора: message-конверты/протоколы,
   schema-валидаторы выходов воркеров, review gates, leases, реестры агентов,
   audit-файлы, "evidence", доменные адаптеры, дашборды.
2. Любой механизм недоверия к воркерам добавляется ТОЛЬКО после реально
   наблюдавшегося инцидента, записанного в `docs/incidents.md` (дата, ран,
   что сломалось). Превентивная защита = bloat.
3. Верификация работы — исполнением: запустить код, прогнать тесты, дать
   критику-воркеру прочитать по существу. Не проверкой формы JSON.
4. Бюджет размера: ядро (`src/`, без тестов) ≤ 1500 строк. Задача, которая
   не влезает — повод остановиться и доложить, а не раздуть.
5. Тесты: поведение механики loop (fake LLM допустим) + live-сценарии.
   ЗАПРЕЩЕНЫ: тесты на наличие строк в промптах, на точные последовательности
   шагов, на существование ключей в dict, тесты-пересказы реализации.

## Архитектура

```
swarm run "task"
  └─ Driver loop (одна LLM: Kimi-K2.6 через Cloud.ru FM API, OpenAI-compatible)
       цикл: chat completion с тулами → исполнить тул → результат в контекст
       тулы (все, других нет):
         spawn_worker(task, worker_id?)      # делегировать подзадачу
         spawn_workers(tasks[])              # параллельная пачка (этап 4)
         run_command(cmd, cwd?)              # исполнить/проверить
         read_file(path) / write_file(path, content)
         finish(result)                      # завершить ран
       остановка: finish() | max_iterations | таймаут рана
```

### Worker contract

- Workspace: `runs/<run_id>/workers/<worker_id>/`
- Харнесс пишет туда `task.md` — текст подзадачи, составленный драйвером.
- Backend `codex_cli` (этап 3): `codex exec` сабпроцессом в workspace,
  sandbox `workspace-write`, прокси из `SWARM_WORKER_PROXY`,
  финальное сообщение через `--output-last-message`.
- Backend `manus` (этап 6): порт адаптера из kaggle_grinder, upstream
  `~/Code/manus-agent`.
- Результат воркера для драйвера: финальное сообщение + список
  созданных/изменённых файлов. Большие выводы — на диск, в контекст путь+TL;DR
  (file-as-memory). Таймаут воркера → kill, в контекст честный отчёт о фейле.

### Run layout и персистентность

```
runs/<run_id>/
  transcript.jsonl    # каждое сообщение и tool-результат, append-only
  result.md           # аргумент finish()
  workers/<id>/       # workspace воркеров
```

Resume = перечитать transcript. Никаких state machine, leases, claims.

### Конфигурация

`.env` в корне (gitignored) + дефолты в коде. Никаких YAML.

| Переменная | Дефолт |
|---|---|
| `FM_API_KEY` | — (обязательна) |
| `SWARM_BASE_URL` | `https://foundation-models.api.cloud.ru/v1` |
| `SWARM_MODEL` | `moonshotai/Kimi-K2.6` |
| `SWARM_MAX_ITERATIONS` | `40` |
| `SWARM_WORKER_TIMEOUT` | `1800` |
| `SWARM_WORKER_PROXY` | пусто (нужен для codex-воркеров) |

## Приёмочные сценарии (live, гоняются с SWARM_LIVE=1)

- **A1 код с самопроверкой**: `swarm run "создай python-пакет strcalc с
  функцией evaluate, evaluate('2+3*4') == 14, с pytest-тестами; прогони тесты"`
  → чекер: в workspace рана pytest проходит. Вмешательств: 0.
- **A2 декомпозиция с зависимостью**: задача, требующая ≥2 воркеров, где
  второй потребляет артефакт первого → чекер: оба workspace непусты, финальный
  артефакт существует и валиден.
- **A3 честная остановка по бюджету**: большая задача при
  `SWARM_MAX_ITERATIONS=3` → ран завершается статусом `budget_exceeded`
  с внятным partial-отчётом, а не зависает и не врёт "completed".

Чекеры — маленькие скрипты, проверяющие реальные артефакты реального прогона.
Smoke-театр (проверка, что сценарий делает то, что написано в сценарии) запрещён.
